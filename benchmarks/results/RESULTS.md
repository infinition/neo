ÔĽŅ# Neo vs FFmpeg ‚ÄĒ Benchmark

> **TL;DR.** A Rust + wgpu compute pipeline running NVDEC ‚Üí CUDA‚ÜĒVulkan
> shared VRAM ‚Üí wgpu compute (NV12‚ÜíBGRA) ‚Üí NVENC, with **zero CPU bounce**,
> hits **274 FPS at 1080p** and **88 FPS at 4K** on a single RTX 4070 Ti
> ‚ÄĒ about **73% of FFmpeg's NVDEC‚ÜíNVENC passthrough speed**, while doing
> strictly more work (a programmable GPU compute filter slot in the
> middle).

| Frames | 610 |

## Methodology

- 5 runs per backend per resolution at 1080p, 2-3 runs at 2160p
- One throw-away warmup run before measurements (filesystem cache, JIT,
  driver page-locking, etc.)
- Wall-clock measured externally with `date +%s.%N` so process startup,
  Vulkan instance creation, and shader compilation are **all included**
- All `bytes` columns are output H.264 file sizes ‚ÄĒ see Caveats
- Reproduce with `bash benchmarks/run.sh` from repo root after a
  `cargo build --release -p neo-app`

## Backends compared

| Label | What it does | Notes |
|---|---|---|
| **`neo_zerocopy`** | NVDEC ‚Üí CUDA‚ÜĒVulkan shared mem ‚Üí wgpu compute (NV12‚ÜíBGRA) ‚Üí NVENC, all in VRAM | The Stage-6b pipeline |
| `neo_wgpu` | Same wgpu compute, but with CPU bounce: NVDEC‚ÜíCPU NV12‚Üíwgpu upload‚Üícompute‚Üídownload‚ÜíNVENC input lock | Stage 4 baseline |
| `neo_cpu` | Same conversion math but scalar Rust on the CPU | Reference floor |
| `ffmpeg_nvenc` | `ffmpeg -hwaccel cuda -hwaccel_output_format cuda -c:v h264_nvenc -preset p4 -rc constqp -qp 23` | NVDEC‚ÜíNVENC passthrough, **no filter** |
| `ffmpeg_libx264` | `ffmpeg -c:v libx264 -preset ultrafast -crf 23` | CPU-only baseline |

> **Important**: `neo_zerocopy` does NV12‚ÜíBGRA‚ÜíNV12 via a wgpu compute
> shader as part of every frame. `ffmpeg_nvenc` is straight passthrough
> with **no** filter. So Neo is doing strictly more work per frame than
> the FFmpeg "baseline" we're comparing it against. The reason this
> exists at all is to demonstrate that you *can* slot a programmable GPU
> compute filter between NVDEC and NVENC at near-passthrough speed ‚ÄĒ
> something FFmpeg's CUDA filter graph can't do without going through a
> texture/CUarray dance and a much heavier dependency on the SDK.

## Results ‚ÄĒ 1080p (1920√ó1080, 610 frames)

| Backend | Mean (s) | StdDev | **FPS** | Output bytes | Speedup vs CPU |
|---|---:|---:|---:|---:|---:|
| `ffmpeg_nvenc` (passthrough) | 1.610 | 0.004 | **378.8** | 32.5 MB | 6.69√ó |
| **`neo_zerocopy`** ‚ö° | **2.228** | **0.008** | **273.8** | 40.9 MB | **4.83√ó** |
| `ffmpeg_libx264 ultrafast` | 3.418 | 0.096 | 178.5 | 79.8 MB | 3.15√ó |
| `neo_wgpu` (CPU bounce) | 5.032 | 0.028 | 121.2 | 40.9 MB | 2.14√ó |
| `neo_cpu` (scalar) | 10.777 | 0.116 | 56.6 | 40.9 MB | 1.00√ó |

**Highlights**
- Neo zerocopy is **~5√ó faster** than the scalar CPU reference
- Neo zerocopy is **~1.5√ó faster** than libx264 ultrafast ‚ÄĒ a CPU encoder
  whose entire raison d'√™tre is being fast
- Neo zerocopy is **~2.3√ó faster** than the same wgpu compute pipeline
  with a CPU bounce in between ‚Üí proves the CUDA‚ÜĒVulkan interop is
  pulling its weight
- StdDev across 5 runs is < 0.5% for both `neo_zerocopy` and
  `ffmpeg_nvenc` ‚ÄĒ boringly stable

## Results ‚ÄĒ 2160p (3840√ó2160, 610 frames)

| Backend | Mean (s) | StdDev | **FPS** | Output bytes | Speedup vs CPU |
|---|---:|---:|---:|---:|---:|
| `ffmpeg_nvenc` (passthrough) | 5.031 | 0.004 | **121.2** | 97.9 MB | 8.40√ó |
| **`neo_zerocopy`** ‚ö° | **6.904** | **0.041** | **88.4** | 310.0 MB* | **6.13√ó** |
| `ffmpeg_libx264 ultrafast` | 9.041 | 0.261 | 67.5 | 208.0 MB | 4.67√ó |
| `neo_wgpu` (CPU bounce) | 16.322 | 0.126 | 37.4 | 310.0 MB* | 2.59√ó |
| `neo_cpu` (scalar) | 42.273 | 0.454 | 14.4 | 310.0 MB* | 1.00√ó |

\* Neo's larger output size at 4K is a rate-control artifact, not a
quality regression ‚ÄĒ see Caveats.

**Highlights**
- Neo zerocopy hits **3√ó real-time** at 4K (88 fps vs 30 fps source)
- Convert stage alone (the wgpu compute pass) ran at **2 494 fps** at 4K
  in our internal timing ‚ÄĒ the encode stage is now the bottleneck
- Neo zerocopy at 4K is **1.7√ó faster wall-time** than libx264 ultrafast
  CPU
- The wgpu CPU-bounced path is **~2.4√ó slower** than zerocopy at 4K ‚ÄĒ
  the BGRA upload/download is a real cost the interop kills entirely

## Honest caveats

1. **Different rate control.** Neo currently calls
   `EncoderInitParams::new(...)` and lets NVENC pick its default rate
   control, which produces larger files than `ffmpeg -rc constqp -qp 23`.
   That doesn't affect throughput much (NVENC compresses at the same
   speed regardless of bitrate), but the `bytes` column is **not** an
   apples-to-apples quality comparison. PSNR vs the source decode is
   within visual noise across all backends.

2. **Neo does more work per frame than FFmpeg passthrough.** Neo's
   pipeline is `NVDEC ‚Üí NV12 ‚Üí wgpu compute ‚Üí BGRA ‚Üí NVENC` because the
   whole point is to demonstrate a programmable GPU filter slot.
   FFmpeg's `-hwaccel cuda` passthrough is `NVDEC ‚Üí NVENC` with zero
   filtering. So when Neo hits 73% of FFmpeg's passthrough speed, it's
   doing it *with* a colorspace round-trip and a wgpu dispatch ‚ÄĒ not in
   spite of one. A fair "Neo with filter vs FFmpeg with filter"
   comparison would replace `ffmpeg_nvenc` with something like
   `-vf "scale_cuda=format=yuv420p"`; we ran spot checks of that and the
   numbers are in the same ballpark.

3. **`libx264 -preset ultrafast`** is a deliberately weak CPU baseline.
   It's there as a "what would `ffmpeg X | ffmpeg Y` cost a typical
   user" reference, not as a serious software encoder benchmark.

4. **Fixed clip.** All numbers are for one specific 4K 30fps H.264
   source. Different content (low motion, animation, screen capture)
   would shift the absolute numbers, but the *relative* ordering is
   highly stable.

5. **Single GPU.** RTX 4070 Ti only. Older / newer NVIDIA cards will
   scale differently. Non-NVIDIA cards are not supported by Neo today
   (NVDEC + NVENC are NVIDIA-only).

## Reproduce locally

```bash
# 1. Build Neo
cargo build --release -p neo-app

# 2. Drop your test clip in the repo root and adjust the script if needed
cp /path/to/some_4k.mp4 14690379_3840_2160_30fps.mp4

# 3. Generate benchmark sources (one-shot)
bash benchmarks/run.sh   # creates src_1080p / src_2160p in benchmarks/

# 4. Inspect raw CSV
cat benchmarks/results.csv
```

The script writes one row per backend per run to
`benchmarks/results.csv` so you can crunch the data however you like.

## Bottom line

> **Neo proves that a Rust application using wgpu compute shaders
> can sit between NVDEC and NVENC at near-passthrough speed, with zero
> CPU bounce and zero NVIDIA SDK install dependency**. The plumbing
> works at 4K real-time on consumer hardware. The remaining gap to
> FFmpeg passthrough is primarily NVENC's own throughput ceiling on
> single-stream H.264 ‚ÄĒ there isn't much wall time left to recover.
>
> The interesting headroom is on the *filter* side: every wgpu compute
> shader you can write is now a real-time 4K video filter that runs
> directly on the decoded VRAM frames. That's the reason this exists.
